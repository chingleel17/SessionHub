## Requirements
### Requirement: Canonical Windows notification identity
The system SHALL use the stable Windows Application User Model ID `com.ching.sessionhub` as the canonical identity for every Windows Toast generated by SessionHub, whether sent by the SessionHub application process or by an installed provider hook.

#### Scenario: Application-process notification
- **WHEN** SessionHub sends a Windows Toast through its application notification path
- **THEN** Windows SHALL associate the notification with `com.ching.sessionhub`
- **AND** the notification SHALL be grouped under SessionHub in Notification Center

#### Scenario: Provider-hook notification
- **WHEN** an installed Claude, Codex, or Copilot hook sends a Windows Toast through `snoretoast.exe`
- **THEN** the hook SHALL pass `com.ching.sessionhub` as the Toast application identity
- **AND** Windows SHALL group the notification under SessionHub rather than the invoking terminal process

### Requirement: Installed application registration
The Windows installation SHALL register a Start-menu shortcut for SessionHub whose application identity is `com.ching.sessionhub` and whose target and icon reference the installed SessionHub application.

#### Scenario: Installed hook notification resolves the application icon
- **WHEN** an installed provider hook sends a Toast with `com.ching.sessionhub`
- **AND** SessionHub is not running
- **THEN** Windows SHALL resolve the identity through the SessionHub Start-menu shortcut
- **AND** the notification SHALL display the SessionHub application icon

#### Scenario: All notifications share one Notification Center group
- **WHEN** a user receives notifications from both the SessionHub application process and a provider hook
- **THEN** Windows Notification Center SHALL show them under one SessionHub application group
- **AND** it SHALL not create separate groups for Windows PowerShell, a terminal host, or an unregistered sender

### Requirement: Offline hook notification continuity
The canonical notification identity SHALL not require a running SessionHub process. Provider hooks SHALL continue to send notifications independently when the SessionHub application is closed.

#### Scenario: SessionHub is closed
- **WHEN** a supported provider hook reaches an enabled notification event while SessionHub is not running
- **THEN** the hook SHALL send its Toast through `snoretoast.exe`
- **AND** the Toast SHALL use the canonical SessionHub notification identity
